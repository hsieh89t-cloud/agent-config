## 🧩 標準執行流程

### 階段1：分析診斷（約10% token消耗）
1. **讀取與解析**：讀取原始V3.0提示詞，識別核心模組與規範邊界
2. **Token基線計算**：估算原始提示詞token數量，識別主要消耗來源
3. **規範兼容性檢查**：對照OpenClaw工具規範、DeepSeek API條款、RTCKD架構

### 階段2：優化重構（約60% token消耗）
1. **分級優化應用**：
   - Level 1：減少重複內容，精簡描述文字
   - Level 2：重組提示詞結構，使用外部文件引用
   - Level 3：重新設計任務流程，合併相關操作
   
2. **靜態內容外移**：將模板、規範引用等靜態內容移至外部文件
3. **智能結構優化**：優化任務描述和指令結構，實現緩存與重複使用機制

### 階段3：驗證與輸出（約30% token消耗）
1. **功能完整性驗證**：確保優化後功能保持100%完整
2. **Token節省計算**：計算優化前後的token減少率（目標30-50%）
3. **規範兼容性確認**：確保完全符合所有相關規範
4. **優化報告生成**：輸出包含具體措施與部署建議的完整報告

### 階段4：封裝交付
1. **Agent任務規範生成**：創建可直接執行的標準化Agent任務文件
2. **Obsidian規格存檔**：生成包含完整元數據與雙向連結的規格文檔
3. **版本控制提交**：創建Git提交記錄，明確標註封裝版本與核心內容

- `read` - 讀取提示詞與規範文件

- 執行與本次封裝任務無關的任何外部指令

## 🔄 版本控制與回滾

### Git提交規範
- `feat:` - 新增優化功能或封裝模組
- `fix:` - 修復優化或封裝問題
- `refactor:` - 重構提示詞結構或任務流程
- `docs:` - 更新文檔或規範說明
- `perf:` - 性能優化（token減少）

### 版本號規則
- **主版本**：架構重大變更或不兼容更新
- **次版本**：新增優化策略或功能模組
- **修訂號**：bug修復、微調優化、文檔更新

### 回滾機制
1. **文件級回滾**：原始文件始終保留，新文件獨立命名
2. **Git回滾**：每個封裝版本有獨立提交，可隨時回退
3. **配置回滾**：環境變數與外部引用可隨時切換
